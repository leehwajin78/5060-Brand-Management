# SRS(Software Requirements Specification) 버전별 컨셉 및 흐름 요약

이 문서는 제품 개발 단계에 따른 SRS 버전별 차이점과 MVP(Minimum Viable Product) 전환 과정을 구조화한 요약본입니다.

## 1. 버전별 컨셉 요약 차트

| 파일 버전 | 문서 Revision | 한 줄 컨셉 | 핵심 성격 | 주요 초점 | 실무 판단 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **SRS_v0_1** | 1.0 | 전체 시스템 초안형 | PRD v0.4.2 기반의 풀스펙 SRS 초안 | 42문항, 10가지 패턴, 마스터 브리프 8섹션, B2B 제안서·강의안 전환 포함 | 제품 비전과 전체 구조를 이해하는 원형 문서 |
| **SRS_v0_2** | 1.1 | 검토 반영 보완형 | v1.0의 누락 요구사항·AC·다이어그램·엔터티 보강 | REQ-FUNC 추가, ANSWER 엔터티, Component·UseCase·ERD·Class Diagram, Traceability 보완 | 개발자에게 전달 가능한 요구사항 정합성 강화본 |
| **SRS_v0_3** | 1.1 | 추적성·검증 강화형 | v0_2와 같은 Revision 1.1 계열. 파일상 별도 개정 번호는 없으나 Traceability Matrix 강조 | PRD–SRS–Test Case 연결, JSONB 저장, Claude Prompt 설정 변경, 요구사항 커버리지 확인 | QA·검수·개발 인수인계용 정교화/검증 관점 문서로 해석 가능 |
| **SRS_v0_4** | 1.3 | Zero-Cost Personal MVP형 | 장기 제품 비전은 유지하되, 개인이 무료 인프라로 구현 가능한 범위로 축소 | 16문항 진단, 리드/답변 저장, AI 진단 리포트, 관리자 검수, 승인 리포트 웹뷰, 상담 CTA | 실제 구축 착수에 가장 적합한 실행형 MVP 문서 |

## 2. 컨셉 변화 흐름

| 흐름 | 변화 방향 | 의미 |
| :--- | :--- | :--- |
| **v0_1 → v0_2** | 초안 → 보완본 | “무엇을 만들 것인가”에서 “요구사항을 빠짐없이 명세했는가”로 이동 |
| **v0_2 → v0_3** | 보완본 → 검증/추적성 강화 | 개발·테스트·검수 관점에서 요구사항 연결성을 강화 |
| **v0_3 → v0_4** | 풀스펙 → 개인 MVP | 상용 제품 완성보다 3~4주 내 무료 파일럿 구현 가능성을 우선 |

## 3. 한 장 요약 (비교 매트릭스)

| 구분 | v0_1 | v0_2 | v0_3 | v0_4 |
| :--- | :--- | :--- | :--- | :--- |
| **제품 방향** | 전체 제품 | 전체 제품 보완 | 전체 제품 검증 | **무료 파일럿 MVP** |
| **구현 난이도** | 높음 | 더 높음 | 더 높음 | **낮춤** |

| **주요 산출물** | 마스터 브리프 | 마스터 브리프 + 보완 요구 | 추적 가능한 SRS | AI 진단 리포트 |
| **질문 범위** | 42문항 중심 | 42문항 중심 | 42문항 중심 | 16문항 중심 |

## 4. 최종 판단

현재 바로 개발에 들어갈 기준 문서는 **SRS_v0_4**가 맞습니다.

다만, v0_1~v0_3는 폐기 대상이 아닙니다. 향후 **V1.5/V2** 단계에서 '마스터 브리프', '원라이너 3종', 'F15 코치 분석 노트', '교차검증 9개 매트릭스'를 복원하고 고도화할 때 반드시 참조해야 할 **상위 설계 자산**으로 관리되어야 합니다.
